58 ET: Legacy (QUAKE3) FILESYSTEM
59 
60 FIXME: This docu needs an update - the following lines aren't fully valid for ET: Legacy
61 
62 All of ET: Legacy's (Quake's) data access is through a hierarchical file system, but the contents of
63 the file system can be transparently merged from several sources.
64 
65 A "qpath" is a reference to game file data. MAX_ZPATH is 256 characters, which must include
66 a terminating zero. "..", "\\", and ":" are explicitly illegal in qpaths to prevent any
67 references outside the quake directory system.
68 
69 The "base path" is the path to the directory holding all the game directories and usually
70 the executable. It defaults to ".", but can be overridden with a "+set fs_basepath c:\etlegacy"
71 command line to allow code debugging in a different directory (make sure it contains an install)
72 Basepath cannot be modified at all after startup and is usually readonly.
73 
74 The "home path" is the path used for all write access. On win32 systems we have had "base path"
75 == "home path", but this no longer the case for ET: Legacy - it requires "base path"
76 != "home path" Any files that are created (demos, screenshots, etc) will be created relative
77 to the base path, so home path should usually be writable.
78 "home path" points to ~/.etlegacy or c:<USER>\My Documets\ETLegacy by default.
79 
80 The user can also install custom mods and content in "home path", so it should be searched
81 along with "home path" and "base path" for game content. Mods downloads from online server games
82 are installed into "home path". These downloads install client files only. A full mod server
83 and client install should be done in "base path". This ensures the install of the game or mods
84 are never overwritten by any downloads.
85 
86 The "base game" is the directory under the paths where data comes from by default, and
87 can be any path if set.
88 
89 The "current game" may be the same as the base game, or it may be the name of another
90 directory under the paths that should be searched for files before looking in the base game.
91 This is the basis for addons.
92 
93 Clients automatically set the game directory after receiving a gamestate from a server,
94 so only servers need to worry about +set fs_game.
95 
96 No other directories outside of the base game and current game will ever be referenced by
97 filesystem functions.
98 
99 To save disk space and speed loading, directory trees can be collapsed into zip files.
100 The files use a ".pk3" extension to prevent users from unzipping them accidentally, but
101 otherwise the are simply normal uncompressed zip files. A game directory can have multiple
102 zip files of the form "pak0.pk3", "pak1.pk3", etc. Zip files are searched in decending order
103 from the highest number to the lowest, and will always take precedence over the filesystem.
104 This allows a pk3 distributed as a patch to override all existing data.
105 
106 Because we will have updated executables freely available online, there is no point to
107 trying to restrict demo / oem versions of the game with code changes. Demo / oem versions
108 should be exactly the same executables as release versions, but with different data that
109 automatically restricts where game media can come from to prevent add-ons from working.
110 
111 File search order: when FS_FOpenFileRead gets called it will go through the fs_searchpaths
112 structure and stop on the first successful hit. fs_searchpaths is built with successive
113 calls to FS_AddGameDirectory
114 
115 Additionaly, we search in several subdirectories:
116 current game is the current mode
117 base game is a variable to allow mods based on other mods
118 (such as etmain + missionpack content combination in a mod for instance)
119 BASEGAME is the hardcoded base game ("etmain")
120 
121 e.g. the qpath "sound/newstuff/test.wav" would be searched for in the following places:
122 
123 home path + current game's zip files
124 home path + current game's directory
125 base path + current game's zip files
126 base path + current game's directory
127 cd path + current game's zip files
128 cd path + current game's directory
129 
130 home path + base game's zip file
131 home path + base game's directory
132 base path + base game's zip file
133 base path + base game's directory
134 cd path + base game's zip file
135 cd path + base game's directory
136 
137 home path + BASEGAME's zip file
138 home path + BASEGAME's directory
139 base path + BASEGAME's zip file
140 base path + BASEGAME's directory
141 cd path + BASEGAME's zip file
142 cd path + BASEGAME's directory
143 
144 server download, to be written to home path + current game's directory
145 
146 
147 The filesystem can be safely shutdown and reinitialized with different
148 basedir / cddir / game combinations, but all other subsystems that rely on it
149 (sound, video) must also be forced to restart.
150 
151 Because the same files are loaded by both the clip model (CM_) and renderer (TR_)

153 load the file with a request to cache. Only one file will be kept cached at a time,

423 {
424  int i = 0;
425  long hash = 0;
426  char letter;
427 
428  while (fname[i] != '\0')
429  {
430  letter = tolower(fname[i]);
431  if (letter == '.')
432  {
433  break; // don't include extension
434  }

436  {
437  letter = '/'; // damn path names

441  letter = '/'; // damn path names

443  }
444  hash += (long)(letter) * (i + 119);

446  }
